1. SYSTEM ARCHITECTURE & DESIGN PHILOSOPHY
--------------------------------------------------------------------------------
The system is built on a "Single Source of Truth" principle using a normalized 
MySQL database. It avoids hardcoded business logic in favor of data-driven 
decisions (e.g., Eligibility Criteria are rows in a table, not `if` statements).

KEY MODULES:

A. AUTHENTICATION & RBAC (Role-Based Access Control)
   - Users are strictly categorized: STUDENT, PLACEMENT_STAFF, ADMIN.
   - Security is handled via stateless JWT (JSON Web Tokens).
   - Middleware (`authMiddleware.js`) intercepts requests to enforce roles.

B. DYNAMIC ELIGIBILITY ENGINE
   - Jobs have linked criteria (`eligibility_criteria` table).
   - When a student views jobs, a SQL query compares their profile (`students` table)
     against the job criteria (CGPA, Semester, Department).
   - This ensures 100% accuracy and prevents ineligible applications at the DB level.

C. RESUME AUDIT SYSTEM (The Core Differentiator)
   - Students can have multiple resumes.
   - Every upload creates a new VERSION (`resume_versions` table).
   - Applications link to a SPECIFIC version ID.
   - LOGIC: If a student edits a resume *after* applying, the `resumeService` 
     detects the active application and sets `resume_updated_after_apply = TRUE`.
   - Staff Dashboard highlights these flagged applications for review.
